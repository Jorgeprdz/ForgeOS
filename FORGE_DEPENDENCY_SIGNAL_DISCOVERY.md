# Forge Dependency Signal Discovery

Status: DISCOVERY

Architecture only. No implementation. No code. No UI. No engines. No schemas.
No Build Tree changes. No ratification. No scores. No rankings. No person
labels.

## 1. Purpose

This document explores evidence that Forge may be creating dependency rather
than professional development.

Dependency is not frequent usage.

Dependency is reduced professional capability caused by repeated outsourcing of
judgment, responsibility or reasoning to Forge.

## 2. Healthy Reliance vs Dependency

Healthy reliance:

- Forge helps the user reason
- Forge shows evidence
- Forge clarifies context
- Forge teaches patterns
- Forge supports reflection
- the user retains responsibility
- the user can explain the decision
- the user becomes more capable over time

Dependency:

- Forge replaces reasoning
- Forge becomes the authority
- Forge hides evidence behind conclusions
- Forge supplies commands
- the user avoids responsibility
- the user cannot explain the decision
- the user becomes less capable without Forge

The distinction is not usage volume. The distinction is capability transfer.

## 3. Advisor Dependency Signals

Potential dependency evidence:

- the advisor asks "What should I do?" repeatedly without asking why
- the advisor cannot explain the reason behind an action
- the advisor sends messages without adapting tone or context
- the advisor stops identifying risks independently
- the advisor waits for Forge before routine follow-ups
- the advisor treats every recommendation as correct by default
- the advisor feels unable to decide when Forge is unavailable
- the advisor requests scripts instead of learning conversation structure
- the advisor uses Forge to avoid difficult client judgment
- the advisor ignores missing context because Forge produced an output

These signals are not proof of dependency. They are design risk signals.

## 4. Manager Dependency Signals

Potential manager dependency evidence:

- the manager uses Forge output as final authority in coaching
- the manager says "Forge says" instead of explaining development reasoning
- the manager stops observing people directly
- the manager delegates hard conversations to system interpretation
- the manager uses Forge to justify promotion, pressure or exclusion
- the manager treats evidence candidates as verdicts
- the manager relies on production or activity without context
- the manager reduces development to compliance with system guidance

Manager dependency is especially dangerous because it can turn Forge from a
coaching aid into a governance substitute.

## 5. System Authority Creep

Authority creep occurs when a tool gradually becomes treated as a source of
truth beyond its rightful boundary.

Examples:

- recommendation becomes command
- evidence candidate becomes proof
- explanation becomes final judgment
- pattern becomes rule
- context becomes pressure
- guidance becomes evaluation
- insight becomes label

Skynet should prevent unsafe intelligence behavior.

Article 0 should prevent the deeper failure: Forge becoming so useful that
humans stop practicing judgment.

## 6. Dependency Loop

A possible dependency loop:

1. User faces ambiguity.
2. Forge gives a useful answer.
3. User acts without understanding the reasoning.
4. Outcome is acceptable.
5. User learns that asking Forge is easier than reasoning.
6. User practices less judgment.
7. Future ambiguity feels harder.
8. User asks Forge for even more direct answers.

This loop can create short-term productivity and long-term professional
weakness.

## 7. False Dependency Signals

Forge should avoid mistaking healthy behavior for dependency.

False dependency signals:

- a new advisor uses Forge heavily while learning
- an advisor asks for help in a genuinely unfamiliar case
- a manager uses Forge evidence to prepare better coaching
- a user asks for examples to understand a principle
- a user requests language help after making the decision
- a user relies on Forge for documentation recall, not judgment

High support need during learning is not automatically dependency.

## 8. Protective Design Concepts

This PAQ does not approve implementation, but future design should consider:

- show evidence before conclusions in high-impact decisions
- make uncertainty visible
- invite the user to explain reasoning in sensitive contexts
- teach principles, not only answers
- offer alternatives with tradeoffs
- make recommendations editable and contextual
- avoid command language when human responsibility is required
- distinguish "suggested" from "decided"
- help managers coach reasoning, not outsource judgment

These are design concepts only, not implementation approval.

## 9. Boundary

Forge must never use dependency signals to shame, rank, punish or label users.

The purpose of detecting dependency is constitutional protection:

to ensure Forge strengthens human judgment instead of replacing it.
