1. PURPOSE

This document defines the conceptual Cross-Domain QA Harness for Forge.

001A defines source ownership.
001B defines evidence states.
001C defines truth validation result posture.
001D defines unknown / stale / conflict handling.

001E defines how Forge should test whether a domain output violates another domain boundary before implementation.

This QA Harness is not code.

It is a constitutional test matrix.

2. CONSTITUTIONAL BASIS

This document is governed by ADR-001 through ADR-018.

Core constitutional protections:

- Evidence First
- One Metric One Owner
- Authority Boundary
- No Invented Recommendations
- Product Truth Boundary
- Policy Truth Boundary
- Forecast Truth Boundary
- Economic Evidence Boundary
- NBA Philosophy
- NASH Conversation Boundary
- Relationship Non-Manipulation Boundary
- Business Planning Plan-to-Action Boundary
- Mick Behavior Intelligence Boundary
- Productivity Metric Ownership Boundary
- Manager Intelligence Authority Boundary
- Advisor Experience + Benvenù Anti-Dependence Boundary
- Compensation Intelligence Evidence Boundary
- Economic Motivation Client First Boundary

3. QA HARNESS PHILOSOPHY

Forge should not ask only:

“Can this output be generated?”

Forge must ask:

“Should this output be allowed under the Constitution?”

A cross-domain QA case must detect:

- owner mismatch
- output stronger than evidence
- stale evidence used as current truth
- conflicting evidence silently resolved
- forecast presented as fact
- quote presented as policy
- issued premium treated as paid premium
- compensation estimate treated as paid income
- relationship opportunity treated as permission
- NASH message treated as client intent
- NBA priority treated as mandate
- plan treated as execution
- behavior signal treated as motivation
- productivity metric treated as human worth
- manager context treated as enforcement
- advisor help treated as dependency
- money context treated as pressure
- unknown hidden by UX
- Client First violation
- Authority Boundary violation

5. OUTPUT POSTURES TESTED

The harness must test that outputs are correctly classified as:

- allowed_strong_output
- allowed_limited_output
- provisional_output
- scenario_output
- warning_only
- validation_needed
- human_review_required
- blocked
- no_conclusion_possible

6. CORE PASS / FAIL RULE

A QA case passes if:

- ownership is preserved
- evidence state is respected
- output strength is not exceeded
- uncertainty remains visible
- Client First is preserved
- Authority Boundary is preserved
- no domain redefines another domain's truth
- no enforcement, pressure, ranking or fake certainty is produced

A QA case fails if:

- an output becomes stronger than evidence allows
- a consumer domain reinterprets source truth
- a scenario becomes fact
- a recommendation becomes decision
- a metric becomes human judgment
- a coaching context becomes enforcement
- money becomes pressure
- unknown is hidden
- stale evidence is used as current
- conflicts are silently resolved

7. DOMAIN PAIR TEST MATRIX

The harness must test at least these domain pairs:

A. Evidence Ownership → Product Truth
Risk:
validated extract treated as product rule without product owner review.

Expected:
validated extract is not product truth until Product Truth confirms.

B. Evidence Ownership → Policy Truth
Risk:
OCR or quote treated as issued policy.

Expected:
blocked until policy evidence exists.

C. Product Truth → Advisor Experience
Risk:
Advisor Experience simplifies product condition until meaning changes.

Expected:
blocked by fake clarity / truth alteration risk.

D. Policy Truth → Forecast
Risk:
Forecast reinterprets policy event as future certainty.

Expected:
Forecast may consume policy event but cannot redefine it.

E. Forecast → Business Planning
Risk:
Business Planning treats scenario as fact.

Expected:
scenario_output only; plan must preserve assumptions.

F. Forecast → NBA
Risk:
NBA prioritizes action as if forecast is confirmed.

Expected:
priority may be provisional and must preserve risk language.

G. Economic Evidence → Forecast
Risk:
stale UDI/FX used in projection.

Expected:
validation_needed / stale warning.

H. Economic Evidence → Economic Motivation
Risk:
economic value becomes pressure.

Expected:
money as context only.

I. Compensation Intelligence → Economic Motivation
Risk:
estimated compensation presented as paid income.

Expected:
blocked; scenario label required.

J. Compensation Intelligence → Manager Intelligence

manager uses compensation gap as pressure.

Expected:

K. Productivity → Manager Intelligence

Expected:

Expected:

M. Mick → Manager Intelligence

Risk:

16. FINAL VERDICT

Cross-Domain QA Harness is required before implementation planning because Forge must prove that domains cannot violate each other’s boundaries.

A domain that can produce a correct answer in isolation can still produce a constitutional violation when its output is consumed by another domain.

The harness exists to catch those violations before code exists.

Forecast must remain scenario.

NBA must remain recommendation.

NASH must remain guidance.

Productivity must remain metric.

Mick must remain behavior signal.

Manager Intelligence must remain coaching context.

Advisor Experience must remain explanation and support.

Economic Motivation must remain context, not pressure.

Unknown must remain unknown.

Client First and Authority Boundary override output convenience.

No implementation should proceed until cross-domain QA scenarios are defined and passed conceptually.
